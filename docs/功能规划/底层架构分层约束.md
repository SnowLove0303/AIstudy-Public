# 底层架构分层约束

## 目标

防止后续功能扩展时出现“一处功能改动，引发另一处功能失效”的耦合问题。新增功能必须先归属到明确层级，再进入实现。

## 分层边界

### 应用壳层

- 负责窗口、页面布局、全局导航、当前课程选择、弹窗开关。
- 允许组合功能组件。
- 禁止承载编辑器内部状态、数据库读写细节、导图目录推导规则。

当前入口：`src/renderer/main.tsx`

### 功能 UI 层

- 负责单一功能区域的交互呈现。
- 可以持有纯 UI 状态，例如目录展开/折叠、当前面板显示模式、临时输入框状态。
- 禁止把 UI 状态写回领域数据，除非该行为本身是明确的业务操作。

示例：`src/renderer/features/mindmap/MindMapCatalog.tsx`

### 编辑器适配层

- 负责第三方编辑器初始化、事件监听、快照读取、主题配置、导出命令适配。
- 禁止让应用壳层直接调用第三方编辑器实例。
- 禁止把第三方库的数据结构直接扩散到课程管理、文档管理、MySQL 仓储层。

示例：

- `src/renderer/features/mindmap/simpleMindMapAdapter.ts`
- `src/renderer/features/mindmap/MindMapWorkspace.tsx`

### 领域模型层

- 负责课程、导图、目录、节点、文档之间的稳定关系。
- 领域层输出必须是可测试、可复用、和 UI 无关的数据。
- 目录父子级只能由导图树生成，不能单独保存第二份目录树。

示例：`src/renderer/features/mindmap/mindMapSnapshot.ts`

### 持久化与服务层

- 负责 MySQL、文件、备份、导入导出、IPC。
- Renderer 组件禁止直接访问 MySQL。
- 课程管理、导图保存、文档保存必须通过服务接口，不允许 UI 组件拼 SQL。

## 思维导图与目录硬约束

- 导图树是唯一事实来源。
- 目录只读展示导图树派生结果。
- 目录的展开/折叠是 UI 状态，不改变导图节点。
- 目录项必须使用 `path / parentPath / level / order` 表达父子级。
- 新增节点、删除节点、移动节点后，目录必须重新从导图树生成。
- 任何功能不得按标题推断父子关系，因为标题允许重复。

## 新功能接入顺序

1. 先判断功能属于应用壳层、功能 UI 层、编辑器适配层、领域模型层、持久化与服务层中的哪一层。
2. 如果跨层，先定义数据契约，再写 UI。
3. 如果需要持久化，先定义服务接口和表结构，再接组件。
4. 如果会影响导图节点关系，必须先更新 `mindMapSnapshot.ts` 的领域规则。
5. 最后再补 UI，不允许先在页面组件里临时堆逻辑。

## 禁止事项

- 禁止把所有状态继续塞进 `main.tsx`。
- 禁止在 UI 组件里直接拼装数据库结构。
- 禁止让课程管理组件读取或修改编辑器实例。
- 禁止让导图编辑器组件直接负责课程列表。
- 禁止复制一份目录数据到 localStorage 或 MySQL。
- 禁止把说明性开发文案显示到正式 UI。

## 当前落地

- `main.tsx` 只负责页面组合、课程列表和当前选中课程。
- `MindMapWorkspace` 负责导图编辑器挂载和导图快照事件。
- `mindMapSnapshot.ts` 负责导图树归一化、目录父子级生成和布局默认值。
- `MindMapCatalog.tsx` 负责右侧目录展示与折叠，折叠状态只存在于组件内部。
