# 导入器模块化规划

## 目标

导入器作为第一个模块化样板，必须同时服务后续控件导入和 MCP 导入。入口可以不同，但解析、预览、匹配、提交必须走同一套导入包协议。

## 开源参考结论

- GitBook 类产品把导入设计为“选择来源 -> 识别格式 -> 预览 -> 导入”，并支持 Markdown、HTML、Word 等多种来源。
- Mammoth.js 的核心思路是把 docx 转成干净语义内容，而不是强行复制 Word 私有样式。
- canvas-editor 社区已有 docx 导入讨论，结论是复杂 Word 样式不能依赖简单转换长期解决，所以 AIstudy 第一版只承诺语义导入，不承诺完整还原版式。

## 当前实现

- 模块目录：`src/renderer/features/importer`。
- 模块 README：`src/renderer/features/importer/README.md`。
- 第一版入口：Word 文档工具栏的“导入”按钮。
- 第一版 UI：独立弹窗，拖拽或选择文件后解析。
- 第一版格式：`.txt`、`.md`、`.markdown`、`.docx`。
- 第一版目标：导入到当前导图节点绑定的 Word 文档。

## 固定协议

所有导入器必须先生成 `ImportPackage`：

- 来源信息：文件名、类型、大小、导入时间。
- 目标信息：课程、导图、节点、目标类型。
- 内容块：标题或段落。
- 摘要：段落数、字数、建议标题。

UI 确认后，导入器只能生成快照并交给已有文档保存路径。导入器不得直接写 MySQL、课程表、导图快照表或 Word 快照表。

## MCP 预留

后续 MCP 只作为另一个入口：

- `import.preview`：提交来源，返回导入预览。
- `import.commit`：提交已确认的导入包。
- `import.status`：查看导入任务结果。

MCP 不得绕开 `ImportPackage`，也不得直接写入数据库。

## 后续扩展顺序

1. 增加追加/覆盖模式。
2. 支持批量文件导入。
3. 支持按导图分支标题自动匹配节点。
4. 支持 XMind 导入导图结构。
5. 接 MCP 入口。
